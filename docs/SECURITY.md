# Security Model

## Assets

- GitHub Copilot entitlement and token.
- Private repository source and metadata.
- ChatGPT-to-Worker bearer token.
- Future Copilot prompts, responses, and session IDs.
- Cloudflare account and deployment configuration.

## Trust boundaries

1. ChatGPT client to public Worker.
2. Worker to private Container beginning in v0.3.
3. Container to GitHub/Copilot services beginning in v0.3.
4. Future repository tools to private GitHub repositories.
5. Logs and receipts to storage.

## v0.2 controls

- Repository is private.
- MCP requires `Authorization: Bearer <AFO_ASK_COPILOT_TOKEN>` unless local non-production development explicitly disables authentication.
- Bearer values are compared through equal-length SHA-256 digests.
- Origin is validated when supplied.
- `Content-Type` must be `application/json`.
- Request bodies are capped at 256,000 bytes.
- Prompts are capped at 20,000 characters.
- Tool names and argument keys are allowlisted.
- Only `ask_copilot` is exposed.
- The v0.2 tool returns a placeholder and never contacts the Container or Copilot.
- Request IDs are generated by the gateway and returned in `X-Request-ID`.
- Logs contain route and protocol metadata but not authorization headers or full prompts.
- A fixed-window in-isolate rate limit defaults to 60 authenticated requests per 60 seconds.
- Errors returned to the client are normalized and contain no stack traces.
- Secret files are ignored by Git.

## Health behavior

`GET /health` is intentionally non-sensitive. It returns only:

- readiness,
- service name,
- service version,
- generic status.

It must not expose whether secrets exist, configured token values, auth mode, Container bindings, GitHub identity, repository allowlists, or deployment internals.

## Known v0.2 limitations

- The in-isolate rate limiter is not a distributed or durable production quota.
- Dependency versions are not pinned yet.
- MCP transport compatibility has not been tested with ChatGPT.
- The Worker has not been deployed live.
- Container startup and secret injection have not been live-tested.
- Copilot SDK and CLI behavior remain unverified and outside v0.2.
- D1 receipts are not implemented.
- Repository strings are metadata only and are not yet checked against an allowlist.

## Required pre-production controls

- Pin and review dependencies.
- Replace or augment the in-isolate limiter with a durable control.
- Add timeout budgets and runtime circuit breaking.
- Add repository allowlists before repository access.
- Add D1 metadata receipts with minimal retention.
- Verify no mutation-capable built-in tools are available.
- Rotate all secrets after any suspected disclosure.
- Restrict production hostname and allowed origins.
- Test denial paths from outside the intended client.
- Build and test the Container before enabling Worker forwarding.

## Logging policy

Do not log:

- Authorization headers,
- bearer token values,
- GitHub tokens,
- runtime shared secrets,
- complete prompts,
- complete model responses,
- private source files,
- session persistence files.

Allowed default metadata:

- request ID,
- route and HTTP method,
- JSON-RPC method,
- tool name,
- response status,
- latency,
- prompt length,
- model identifier when supplied,
- opaque session handle when supplied,
- redacted error category.

## Incident response

1. Disable the Worker route.
2. Rotate `AFO_ASK_COPILOT_TOKEN`.
3. Rotate `RUNTIME_SHARED_SECRET` when configured.
4. Revoke and replace `COPILOT_GITHUB_TOKEN` when configured.
5. Review Cloudflare and GitHub audit logs.
6. Preserve redacted receipts.
7. Fix and verify locally.
8. Redeploy only with explicit approval.
9. Verify the live endpoint.
10. Create and link a CairnStone incident/recovery stone.
